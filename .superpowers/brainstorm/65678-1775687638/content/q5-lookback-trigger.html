What triggers a lookback check?

Every conviction in the ledger is pending until something checks it against reality. What fires that check?

A

Nightly scheduled job

A background task runs every night at market close. It scans all pending convictions, checks which lookback windows have elapsed (30d, 60d, 90d), fetches the current price via yfinance, and marks them confirmed or contradicted.


⏰ 4:30 PM ET daily
→ scan conviction_ledger.json
→ for each pending: days_elapsed ≥ 30/60/90?
→ fetch price via yfinance
→ mark CONFIRMED / CONTRADICTED
→ update accuracy scores

Why this works: Simple, reliable, no user action needed. Works even when RAPHI isn't being used actively. Accuracy scores are always fresh by morning.

Limit: Requires the server to be running at 4:30 PM. Misses days the server is off.

B

Lazy — check on access

No background job. When the user opens the Conviction Ledger page or a memo with inline accuracy, RAPHI checks all pending convictions whose windows have elapsed at that moment. Results are cached for the session.


User opens Ledger page
→ check all pending convictions
→ elapsed? → fetch price → resolve
→ render accuracy scores
→ cache until next page load

Why this works: Zero infrastructure. No scheduled tasks. No server uptime requirement. Results always match what the user is looking at.

Limit: First page load after a gap feels slow (multiple yfinance calls). Accuracy scores can be stale if user hasn't opened RAPHI in weeks.

C

Event-driven — on dashboard refresh

The existing liveRefresh() cycle already runs every 30 seconds when market is open and every minute when closed. Add conviction lookback as one more task in that Promise.all. Only checks convictions whose windows have elapsed.


liveRefresh() → Promise.all([
  loadMarketMetrics(),
  loadDashboardSignals(),
  loadWatchlistPrices(),
  checkPendingConvictions() ← new
])

Why this works: Fits perfectly into the existing refresh architecture — no new scheduling infrastructure. Convictions resolve in the background as the dashboard updates. Inline accuracy scores stay live.

Limit: Only runs when the dashboard is open. Convictions accumulate if server/UI is idle for days.

My recommendation

C first, A later. Plugging into the existing liveRefresh() cycle is zero new infrastructure and fits the current architecture exactly. It handles the active-user case perfectly. When RAPHI moves to a more production deployment, add the nightly scheduled job (A) as a second trigger for the idle case. Start with C — it's already wired.
